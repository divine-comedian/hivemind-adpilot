## Fixed in this audit

### P1-1 — `.env.example` was missing most required vars  ✅ fixed in `dcbacd5`

Only had `HIVEMIND_*` and `LI_DEFAULT_CAMPAIGN_ID` / `FB_DEFAULT_ADSET_ID`. A fresh checkout would not run — `scripts/*.py` need `LINKEDIN_ACCESS_TOKEN`, `FACEBOOK_ACCESS_TOKEN`, `FACEBOOK_PAGE_ID`, `OPENAI_API_KEY` at minimum.

### P1-2 — Server restart silently breaks the workspace  ✅ fixed in `a69f038`

Onboarding wrote tokens to `workspace/.tokens.env` and `os.environ`, but `.tokens.env` was never read back. After any `uvicorn` restart, every push/analytics/diagnose request would fail with "token not set" until the user re-onboarded — losing the workspace state.

Fixed with a FastAPI `lifespan` hook in `server/main.py` that re-populates `os.environ` from the tokens file on startup.

### P1-3 — Path traversal in `/api/image`  ✅ fixed in `195c6a4`

The check was `p.includes("/workspace/")` — a substring match. A path like `/home/mitch/github/aurevon-ads/workspace/../../../etc/passwd` passes the check, then `readFile` resolves the `..` out of scope and serves arbitrary files. Localhost-only exposure but trivially exploitable if the dev server is ever bound to a non-loopback interface.

Fixed by resolving the absolute path and requiring it to start with the actual workspace directory.

### P1-4 — `api.pushDraft` type signature missing `adset_id`  ✅ fixed in `195c6a4`

The TypeScript type of `pushDraft` only accepted `campaign_id`. The Facebook branch on the server requires `adset_id`. If anything beyond the demo's env-fallback path ever needs to pass an explicit adset, the frontend can't type it. Also typed the return shape concretely instead of `unknown`.

### P2-1 — Duplicated env-setup boilerplate across 3 routes  ✅ fixed in `a69f038`

`drafts.py`, `analytics.py`, and `diagnose.py` each carried the same 3-line `os.environ.setdefault("LINKEDIN_ACCESS_TOKEN", os.environ["LINKEDIN_TOKEN"])` dance. Removed entirely — the startup loader + canonical env names in `workspace.py` POST handle it once.

### P2-2 — `LINKEDIN_TOKEN` / `FACEBOOK_TOKEN` indirection was needless  ✅ fixed in `a69f038`

Onboarding wrote tokens as `LINKEDIN_TOKEN` / `FACEBOOK_TOKEN` (one name) but `scripts/config.py` reads `LINKEDIN_ACCESS_TOKEN` / `FACEBOOK_ACCESS_TOKEN` (another name) — forcing every route to do an indirection dance. Now writes the canonical names directly.

### P2-3 — Dead `token_ref` field in workspace state  ✅ fixed in `a69f038`

`platforms.linkedin.token_ref: "LINKEDIN_TOKEN"` was stored but never read anywhere. Removed.

### P2-4 — Late `pydantic` import in `workspace.py`  ✅ fixed in `a69f038`

`from pydantic import BaseModel as _PydBaseModel` was at line 98 of the file, aliased to avoid shadowing — but there was no shadowing risk. Moved to the top with `from pydantic import BaseModel`. Cleaner.

### P2-5 — `drafts/` and `samples/` weren't in `.gitignore`  ✅ fixed in `dcbacd5`

The pre-existing `drafts/2026-*/` directories total 112MB of generated PNGs that are checked into git already. The gitignore now blocks future generations from being tracked (but doesn't untrack existing files — that's a separate cleanup). `samples/` (47MB) added too.

## Deferred — not blocking, listed for future work

### P2-6 — `prewarm_aurevon.py` couples `scripts/` to `server/`

`scripts/prewarm_aurevon.py` imports `from server.hivemind.client import HivemindClient`. This reverses the original layering (server reuses scripts, not the other way). If `server/` ever becomes a separate package, this dependency breaks. Low impact for the hackathon.

### P2-7 — Tracked 112MB of generated drafts under `drafts/2026-*`

The existing `drafts/2026-*/` directories already in git represent ~17 ad rotation experiments worth of PNG files. Not worth purging during the hackathon (`git filter-repo` would rewrite history). After the demo, consider a one-time cleanup or moving them to git-lfs.

### P3-1 — `Aurevon`-hardcoded brand assets

`scripts/li_campaign.py` uses a hardcoded `_ORG_URN` for Aurevon. `scripts/generate_image.py` references Aurevon logos and brand palette. Onboarding form collects `org_urn`, brand color, and logo upload, but those values aren't piped through to the scripts. Acceptable for the demo (which runs against Aurevon's real account) but the "bring your business" framing is partly aspirational at this layer. Address in any future productionization.

### P3-2 — Image generation style hardcoded to `style_id=1`

Every Generate call uses `style_id=1`. All drafts look visually similar. For variety in the demo, pass a random style id from `scripts/generate_image.py:list_styles()`. ~5 min change if it matters for the recorded Loom.

### P3-3 — `pushDraft` API client doesn't expose explicit `campaign_id`/`adset_id` to the UI

The DraftCard always relies on the env fallback. A more product-shaped flow would let the user pick a campaign/adset on push. Out of scope for the hackathon.

## Architecture notes (no fix, just observations)

- **`scripts/fb_campaign.py` is 723 lines, `scripts/generate_image.py` is 535 lines** — both god-file territory by typical thresholds, but they're pre-existing and out of scope. Don't touch them.
- **`scripts/`-as-library** is a useful pattern: every existing CLI script also exposes its core function (`fetch_analytics`, `create_ad`, etc.), which is why AdPilot could wrap them without rewrites. Worth preserving.
- **No tests at the route layer.** The HivemindClient and chains have unit tests with mocked HTTP, but `server/routes/*.py` are exercised only via manual smoke. For a hackathon this is fine; for production add `TestClient`-based route tests.
- **The `/api/v1/chat` shape assumption** (persona as JSON body field) is still unverified — `server/hivemind/client.py:chat()` carries a docstring noting this. Verify before recording the demo.
